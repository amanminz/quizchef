# ADR-006

Decision: Live gameplay is server-authoritative. The execution engine is the sole authority for progression, timing, answer acceptance, correctness, and scoring.
Status: Accepted
Reason: In a live quiz the incentives to cheat converge on the client — a fast clock, a forged score, a "correct" flag flipped in the browser. If any of those are trusted, the leaderboard is meaningless. So the server owns all game state and derives everything itself: it decides which question is open and when, stamps every answer's response time from its own Clock, determines correctness by comparing submitted option ids to the question's correct set, and computes every score. Clients render state, submit commands, and receive projections — they never compute correctness or points, and the server never returns a score in an answer acknowledgement. Clients may render optimistically for responsiveness — highlighting a tapped answer the instant it is tapped — but the server remains the sole authority: any such prediction is provisional and must reconcile to server state (the accepted/rejected acknowledgement, then the revealed answer and leaderboard). Optimistic UI is a rendering convenience, never a source of truth.
Consequences: Question progression, the QUESTION_OPEN → QUESTION_CLOSED → ANSWER_REVEALED → LEADERBOARD phase machine, and answer acceptance are enforced by the Session aggregate; illegal transitions and late, early, duplicate, or ill-formed answers are rejected. Timing is derived from the shared Clock (ADR-005's in-process determinism extends to gameplay), never System.currentTimeMillis(), so scoring is reproducible and tests are not flaky; a server-side scheduler closes an expired question, and the close is idempotent against a manual host close. Scoring is a framework-independent domain service driven by a ScoringPolicy value object, so scoring variants (Classic, Kids, Practice, Tournament) are a policy swap, not an engine change. The leaderboard is always a projection, never persisted — recomputed from the participants' cached scores. Builds on [ADR-003](ADR-003-durable-participants.md) (durable participants whose score survives reconnection), [ADR-004](ADR-004-transport-independence.md) (the engine emits domain events; the transport projects them), and [ADR-005](ADR-005-application-services-drive-business-operations.md) (host and participant commands enter through application services).
