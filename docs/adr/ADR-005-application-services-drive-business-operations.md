# ADR-005

Decision: Business operations are driven through application services.
Status: Accepted
Reason: Formalizes the rule the architecture already follows. Every state change needs exactly one kind of owner; when controllers, transports, or subscribers can also mutate state, business rules scatter and transactions become untraceable. With a single entry point per operation, the flow is always transport → application service → domain → domain event → subscribers.
Consequences: Only application services may mutate aggregates, publish domain events, and open transactions. Everything else is read-only. Controllers remain thin (validate, delegate, respond). Repositories remain persistence only. The WebSocket module remains transport only: inbound messages delegate to application services, outbound updates arrive as domain event subscriptions — it never orchestrates business logic. Event subscribers perform delivery and side effects, never business operations. The domain event model is framework independent (own DomainEvent contract in the common module, not Spring's ApplicationEvent), keeping the domain portable; the in-process event dispatcher makes the application reactive without Kafka, RabbitMQ, or Spring Cloud, and keeps the session timeline reconstructable (analytics, replay, moderation) without event sourcing. Builds on [ADR-003](ADR-003-durable-participants.md) and [ADR-004](ADR-004-transport-independence.md).
